# План на завтра

Контекст: `PRODUCT_PLAN.md` — общая стратегия, этот файл — конкретный
список дел на следующую сессию, по приоритету.

## Что сделано и задеплоено в prod (2026-07-14)

- **Line-span extraction для telefonnotiz** (`line_extraction.py`) —
  вместо перепечатки текста модель возвращает `{start_line, end_line}`
  в пронумерованный markdown, код сам вырезает текст. Устраняет класс
  багов "модель обрезает/перефразирует поле при перепечатке".
  Дополнительно обнаружен и обработан отдельный случай — один
  ответ-блок, общий сразу для нескольких изданий документа (значения
  через "/" в порядке изданий) — для него добавлено поле `slash_index`.
  Проверено на реальных данных живыми вызовами Gemini.
- **PDF-подсветка → маркер "– 100%"** (`answer_markers.py`), встроено в
  `/api/convert` — автоматически дописывает маркер к найденным по
  жёлтой заливке в PDF правильным ответам, для ВСЕХ типов секций.
  Закрывает ~8%-й wrong-answer-key gap у beschwerde/lesen_teil4
  (типы вопросов без текстового маркера в источнике).
- **Решён блокер размера Vercel-функции**: PyMuPDF (64MB, нужен для
  answer_markers) поверх markitdown[pdf] (203MB) превышал лимит
  Vercel (~250MB). Разделение на отдельные serverless-функции не
  сработало без рефакторинга роутинга (Vercel zero-config Flask не
  поддерживает изолированные per-function зависимости в этом проекте
  — проверено spike-тестом). Вместо этого убрали сам markitdown: его
  PDF-конвертация — это буквально один вызов
  `pdfminer.high_level.extract_text()`, всё остальное (magika +
  onnxruntime, ~126MB) — угадывание типа файла, который и так уже
  известен (`/api/convert` сохраняет файл с суффиксом `.pdf`
  специально для этого вызова). Эквивалентность вывода проверена
  побайтово на реальном PDF перед переключением. Итоговый размер
  задеплоенной функции упал с ~83MB до ~52MB.
- **Multi-run consensus для `verify_content.py`** (`--runs N`, по
  умолчанию 3) — один прогон LLM-аудита ненадёжен (подтверждено:
  два прогона на идентичном входе дали разные результаты).
  Агрегирует находки по кросс-run согласию.
- **`curation_checklist.py`** — оркестратор, связывающий
  `update_curated_content.py` + `check_answer_keys.py` +
  `check_verbatim_content.py` + `verify_content.py` в один
  консолидированный отчёт по diff-based review scope. Ничего не
  патчит автоматически.
- `_parseCacheVersion` в клиенте забампан (v33→v34).
- Прод после деплоя проверен smoke-тестом (OPTIONS/POST отвечают
  корректно, функция стартует без ошибок импорта).

## Что осталось

- [ ] **Promptfoo-регрессия для telefonnotiz line-span/slash_index** —
  есть фикстуры для discover-правил, но не для нового формата промпта
  telefonnotiz. Без теста будущие правки промпта могут незаметно
  сломать line-span/slash_index.
- [ ] **Реальный тест сентинела "нет буллетов" (`start_line=-1,
  end_line=-1`)** — проверялся только синтетически (руками собранный
  fake_parsed), не живым вызовом Gemini на документе, где у издания
  действительно нет weitere_informationen.
- [ ] **lesen_teil2 / hoeren_teil4 ещё не переведены на line-span** —
  паттерн пока работает только для telefonnotiz; для остальных
  verbatim-полей всё ещё перепечатка текста (тот же класс багов
  теоретически возможен там же).
- [ ] **`curation_checklist.py` не встроен как обязательный гейт** —
  существует и работает, но запускается вручную, не автоматически при
  каждом обновлении контента.
- [ ] **Regex-извлечение ключей ответов** по маркеру "– 100%"/"(100%)"
  без LLM — не делали, пока не понадобилось (answer_markers закрыл
  практическую часть проблемы).

## Мелкое, не блокирующее

- Известная неоднозначность источника: у telefonnotiz variant 2,
  издание 2, поле `telefonnummer` в печатном источнике неоднозначно
  (три разных числа претендуют на то же поле в разных местах
  документа). Задокументировано, не регрессия сегодняшней работы,
  осознанно не трогали — `telefonnummer` остаётся plain-text полем
  (не переведено на line-span).
- `DISCOVERY_BUG_ANALYSIS.md` и `scripts/fix_course.py` — намеренно
  untracked (встраивают дословные фрагменты копирайтного источника),
