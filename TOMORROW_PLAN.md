# План на следующую сессию

Контекст: `PRODUCT_PLAN.md` — общая стратегия, этот файл — конкретный
список дел, по приоритету.

## Что сделано и задеплоено в prod (2026-07-14)

- **Line-span extraction для telefonnotiz** (`line_extraction.py`),
  включая `slash_index` для общих ответ-блоков между изданиями.
  Проверено на реальных данных живыми вызовами Gemini.
- **PDF-подсветка → маркер "– 100%"** (`answer_markers.py`), встроено
  в `/api/convert` для всех типов секций.
- **Решён блокер размера Vercel-функции**: markitdown заменён на
  прямой вызов `pdfminer.high_level.extract_text()` (побайтово
  идентичный вывод) — освободило место под PyMuPDF без разделения на
  функции. Итоговый размер функции упал с ~83MB до ~52MB.
- **Multi-run consensus для `verify_content.py`** (`--runs N`) +
  **`curation_checklist.py`** — оркестратор всех 4 инструментов
  сверки, задокументирован в `PRODUCT_PLAN.md` как обязательный шаг
  перед публикацией курируемого контента в Redis.
- **Promptfoo-регрессии для telefonnotiz** — 2 новых живых теста
  (shared-answer-block/slash_index, no-bullets sentinel), оба
  проходят. По пути починен реальный баг в `prompt_parse.py` (не
  нумеровал markdown для telefonnotiz).
- `_parseCacheVersion` в клиенте забампан (v33→v34).

## Что осталось

- [ ] **hoeren_teil4 теряет издания — подтверждённый живой баг.**
  С байт-в-байт тем же входом, что видит прод (форм-фиды markitdown
  на месте): 3/3 прямых вызова вернули только 1 из 3 напечатанных в
  источнике изданий. `hoeren_teil4` уже в `HEAVY_SECTION_TYPES`
  (thinkingLevel LOW) — недостаточно. Нужно отдельное расследование:
  попробовать thinkingLevel MEDIUM, более сильную модель (как у
  discover), или line-span/сегментационные правки промпта. См.
  PRODUCT_PLAN.md, раздел "⚠️ Открытая находка".
- [ ] **promptfoo file:// fixture loader теряет `\x0c`** (form feed) —
  из-за этого `hoeren_teil1`-тест был ложно-красным (сама модель
  работает корректно на реальном, prod-идентичном входе). Нужно
  почистить/задокументировать в самом харнессе, иначе тест будет
  постоянно шуметь.
- [ ] **lesen_teil2 / hoeren_teil4 line-span** — сознательно отложено:
  11 типов вопросов делят одну `_UNIVERSAL`-схему/промпт, поле для
  переноса (`texts[].content`) — многоабзацный форматированный текст,
  не одна буллета как у telefonnotiz. Отдельная, более крупная задача
  — планировать заново, не как продолжение telefonnotiz-паттерна.
- [ ] **Regex-извлечение ключей ответов** по маркеру "– 100%"/"(100%)"
  без LLM — низкий приоритет, answer_markers закрыл практическую часть.
