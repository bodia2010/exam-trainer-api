### 1. Назначение продукта

Exam Trainer — Android-first Flutter-приложение для подготовки к языковому
экзамену. Пользователь выбирает профиль экзамена, загружает собственный PDF с
экзаменационными материалами, а система превращает его в структурированный
курс с интерактивными упражнениями. Приложение не продаёт и не поставляет
чужой экзаменационный PDF: исходный материал приносит сам пользователь.

Текущая реально поддерживаемая комбинация — **telc Beruf B2**. В интерфейсе
уже перечислены будущие профили telc/Goethe, B1/B2/C1, Allgemein/Pflege, но они
помечены как недоступные: для них ещё нет отдельных discovery-промптов, схем и
проверенных тренажёров. UI локализован на немецкий, английский, русский и
украинский языки; язык по умолчанию — немецкий.

Кроме импортируемых курсов есть отдельный встроенный модуль устной части
`Sprechen B2 Beruf` с фиксированным банком заданий для Teil 1, Small Talk
(Teil 2) и Teil 3. Он не зависит от PDF, Gemini или кэша импорта.

### 2. Основной пользовательский путь

1. **Запуск.** Flutter сразу рисует брендированный bootstrap-экран, пока
   локально инициализируется Firebase. На Android 12+ используется отдельная
   splash-тема, поэтому холодный старт не должен показывать пустой чёрный
   экран.
2. **Вход.** Пользователь входит через Google или email/password. При новой
   регистрации он должен принять privacy policy и terms. Firebase Auth
   хранит сессию и выдаёт ID token для backend-запросов.
3. **Проверка устройства.** После входа приложение один раз за сессию вызывает
   `/api/device`. На аккаунт разрешено до двух устройств; экран лимита может
   удалить прежние регистрации и оставить текущее устройство. Сетевой сбой
   здесь fail-open и не блокирует пользователя.
4. **Главный экран.** Пользователь видит импортированные курсы, вход в модуль
   устной части, избранное, настройки языка и аккаунта. Курсы сначала читаются
   локально, затем best-effort дополняются курсами того же UID из Firestore.
5. **Импорт PDF.** Пользователь выбирает доступный профиль экзамена и файл.
   Приложение показывает этапы конвертации, проверки кэша, discovery,
   пораздельного parse и сохранения.
6. **Курс.** Результат сохраняется как `ParsedCourse`: UUID, имя исходного
   файла, дата импорта, профиль экзамена и словарь из 12 типов разделов со
   списками вариантов. Экран курса ведёт к разделу, варианту или режиму
   пробного экзамена.
7. **Тренировка.** Экран показывает исходный текст/диалог, вопрос и варианты
   ответа. Для слуховых заданий можно синтезировать немецкое аудио. Упражнение
   можно добавить в избранное.
8. **Удаление.** Удаление курса сразу очищает локальный JSON и связанные
   bookmarks, затем асинхронно удаляет облачную копию. Удаление аккаунта
   сначала очищает Firestore, затем Firebase Auth и только после подтверждения
   сервера удаляет локальные данные.

### 4. Пайплайн PDF → тренажёр

#### 4.1 Convert

Клиент отправляет бинарный PDF на `POST /api/convert`. Backend временно пишет
его в файл, извлекает текст через `pdfminer`, нормализует переводы строк и
удаляет временный файл в `finally`. Исходный PDF на сервере постоянно не
хранится.

Затем PyMuPDF читает визуальную разметку PDF. Если вариант ответа выделен
жёлтым прямоугольником, `answer_markers.py` находит соответствующую строку
`a)`/`b)`/`c)` в извлечённом тексте и добавляет маркер `– 100%`. Так ключ,
который иначе потерялся бы вместе с форматированием PDF, становится доступен
детерминированно. Этот post-pass best-effort: его сбой не должен ломать саму
конвертацию.

#### 4.2 Whole-document cache

Клиент вычисляет SHA-256 от точного конвертированного текста и сначала
проверяет doc-кэш `v30.v36|doc|<hash>`. Hit возвращает готовые 12 разделов и
полностью пропускает discovery/parse. Ключ чувствителен к любому изменению
текста или алгоритма convert; поэтому курируемый кэш мигрируется после
релевантного version bump.

#### 4.3 Discovery

При doc-cache miss клиент нумерует все строки документа и проверяет discovery
кэш `v30|discover|<hash>`. На miss весь документ один раз обрабатывает
`gemini-3.5-flash` при temperature 0. Модель возвращает только структуру:
`section_type`, `variant_number`, `version_label`, `start_line`, `anchor`.

`anchor` позволяет клиенту исправить потерянную цифру в `start_line`: строка
сверяется с реальным документом, а при расхождении ищется ближайшее совпадение.
Записи `other` обозначают оглавления, устную часть и прочие блоки, которые не
должны становиться упражнениями, но нужны как безопасные границы чанков.

#### 4.4 Grouping и parse

Клиент режет документ по discovery-границам и группирует все редакции одного
`(section_type, variant_number)` через `<<<ITEM>>>`. Каждый group сначала
ищется в кэше `v36|group|<hash>`. Cache miss отправляется на `/api/parse`; до
шести групп обрабатываются параллельно.

Parse использует `gemini-3.1-flash-lite`, per-type prompt и обязательный
Gemini `responseSchema`. Схема фиксирует поля, допустимые типы вопросов и
точное число вопросов там, где формат экзамена это определяет. Клиент после
ответа раскрывает dedup-сентинелы, проверяет структуру и ключи, повторяет
структурно плохой ответ и не кэширует невалидный результат. Редакции одного
варианта объединяются, но разные version labels сохраняются отдельно.

Для verbatim-полей модель не должна перепечатывать длинный исходный текст:

- `telefonnotiz` возвращает line-span для дополнительных сведений;
- `lesen_teil2` и `hoeren_teil4` возвращают line-span для `texts[].content`;
- backend механически вырезает указанные строки из исходного чанка,
  сохраняет абзацы/hyphen-wrap и только затем возвращает обычный контракт
  `{title, content}` клиенту.

Это уменьшает output-токены и делает перефразирование этих полей структурно
невозможным. Line-span не решает отдельную проблему потери целых редакций при
parse — она остаётся самостоятельной задачей.

#### 4.5 Сохранение результата

Полностью успешный Premium-импорт best-effort записывает assembled course в
doc-кэш. Сам курс всегда сначала сохраняется локально в
`documents/courses/<uid>/<courseId>.json`; облачная загрузка не блокирует UI.
Firestore хранит курс одним JSON-полем. Значения больше 900 КБ не отправляются
в облако из-за лимита документа Firestore, но продолжают работать локально.

### 6. Free и Premium

| Возможность | Free | Premium |
|---|---|---|
| Уже обработанный PDF | Да, через общий кэш | Да |
| Новый неизвестный PDF | Нет, понятный 403 | Да, до 5 новых документов/день |
| Варианты раздела | Первый вариант | Все найденные варианты и редакции |
| Telefonnotiz editions | Первая редакция | Все редакции |
| Parse-защита | До 12 000 символов и 8 редакций на group | Полный group |

Free-клиент при doc-cache hit не получает полный Premium-курс: результат
сокращается до первого top-level item каждого раздела, а список `versions` у
telefonnotiz — до первой редакции. Backend независимо повторно ограничивает
размер free parse, поэтому модифицированный клиент не обходит тариф.

Premium сейчас является boolean `users/{uid}.isPremium` в Firestore. В коде
нет Play Billing, подписки или экрана покупки; выдача Premium пока является
внешней административной операцией. Ошибка чтения тарифа всегда трактуется
как Free, а не как неявный Premium.

### 7. Аутентификация, данные и безопасность

- Каждый защищённый запрос несёт `Authorization: Bearer <Firebase ID token>`.
  Backend проверяет RS256-подпись, issuer и audience по Google JWKS и получает
  UID; общего секрета внутри APK нет.
- Локальные курсы и preferences namespaced по UID, поэтому смена аккаунта на
  одном телефоне не раскрывает курсы предыдущего пользователя.
- Firestore содержит `users/{uid}`, `devices`, `courses`; `progress`
  зарезервирован и пока не записывается. Backend обращается к Firestore REST
  через service account, чтобы не тащить тяжёлый Firebase Admin SDK в Vercel.
- Upstash-кэш глобальный и не привязан к UID, потому что одинаковый исходный
  текст должен давать одинаковый parse. В кэше хранится JSON результата, а не
  raw PDF. Rate-limit и дневные import caps также живут в Redis с TTL.
- Лимиты: 1000 API-запросов/час на UID, 5 новых discovery/день на Premium и
  100 новых discovery/день на весь сервис. Кэш-хиты не расходуют Gemini.
- Избранное хранится локально. При удалении курса связанные bookmarks
  удаляются каскадно, чтобы не оставлять мёртвые маршруты.
- Account deletion удаляет подколлекции `devices/courses/progress`, корневой
  документ пользователя и затем Firebase Auth account. Частичный сбой после
  удаления данных отдельно сообщается клиенту. Ручной E2E на физическом
  Android-устройстве пройден 2026-07-15: production вернул полный успех,
  приложение перешло на экран входа, локальные файлы курсов и scoped
  preferences отсутствуют. Это также подтверждает рабочее IAM-разрешение
  service account на удаление Firebase Auth user.

### 8. Аудио

Слуховые упражнения используют `/api/tts`. Клиент разбивает диалог по
говорящим и длинные реплики — на фрагменты до 400 символов. Backend выбирает
немецкий мужской/женский голос детерминированно по имени говорящего и
синтезирует MP3 через Microsoft Edge TTS. Клиент кэширует каждый фрагмент на
диске по хэшу, поэтому повторное прослушивание не требует нового запроса.

## Фаза 0 — защита кошелька (СНАЧАЛА, до публикации; ~полдня)

Сейчас rate limit — 1000 запросов/час на пользователя. Один discover
≈ $0.35 → злонамеренный аккаунт с уникальными PDF может сжечь сотни
долларов за час. Оба тарифа на платных ключах, поэтому:

### 0.1 Лимит импортов (discover-вызовов) в день на пользователя
- В `main.py::parse()` для `section_type == 'discover'` (только при
  кэш-промахе — кэш-хиты не считаются): Redis-счётчик
  `importcap|{uid}|{YYYYMMDD}`, expire 24ч.
- Premium: **5 новых документов/день** (реальному студенту хватает с
  запасом; потолок расходов $2.5/день на аккаунт).
- Free: см. 0.2.
- Ответ при превышении: 429 + понятное сообщение (клиент уже умеет
  показывать серверные ошибки лимитов).

### 0.2 Free-тариф: discover только по кэшу (рекомендуемый вариант)
Free-пользователь НЕ может запустить discover для документа, которого
нет в кэше — вместо этого получает сообщение «этот документ ещё не
обработан — доступно в Premium». Следствия:
- Free-тариф теперь стоит **строго $0** в Gemini-расходах на discover
  (parse для free и так ограничен 12K символов / 8 изданий — копейки,
  плюс тоже можно закрыть той же логикой «только по кэшу»).
- Популярную компиляцию зальёт первый же premium (или мы сами через
  Фазу 1) — free-пользователи получат полный опыт на главном контенте.
- Незнакомый PDF становится апселлом в premium — честная и понятная
  граница тарифа.
- Реализация: в `parse()` при кэш-промахе и `not premium` → 402/403 с
  кодом ошибки для клиента. Альтернатива (если вариант покажется
  жёстким): 1 импорт/день free — но это до $0.5/день на каждый free-
  аккаунт, при масштабе это дыра; вариант «только по кэшу» безопаснее.

### 0.3 Глобальный дневной предохранитель
Redis-счётчик discover-вызовов на весь сервис в день (например, 100).
При превышении новые документы получают «сервис перегружен, попробуйте
завтра», кэш-хиты продолжают работать. Защищает от координированного
злоупотребления множеством аккаунтов. Констант в одном месте, рядом с
`_RATE_LIMIT_PER_HOUR`.

### 0.4 Мониторинг расходов
- Еженедельно: `vercel logs ... | python3 scripts/cost_report.py`
  (уже считает по-модельно честно).
- В GEMINI_USAGE лог добавить `uid=` (усечённый) — чтобы при аномалии
  видеть, кто именно жжёт токены.

**Definition of done Фазы 0:** злонамеренный аккаунт любого тарифа не
может потратить больше $2.5/день; free-тариф стоит ~$0; есть глобальный
стоп-кран.

### 2.3 Извлечение ключей из PDF-разметки (highlights/underline)

маркера «– 100%» нет. MarkItDown выбрасывает форматирование, поэтому в

rect'ы (fill 1,1,0, высота >5pt) поверх строк опций; `page.get_text

строке-опции = ключ ответа; жёлтый на заголовке = просто выделение
заголовка; зелёный = мета-метки («Новый вариант от ...») — фильтровать
по паттерну строки-опции `^[a-c]\)`.

1. **Офлайн сразу** (СДЕЛАНО 2026-07-12): `scripts/check_answer_keys.py`

### 2.5 Известная проблема: промпт заставляет галлюцинировать, когда источник не даёт ответа

источнике для этого издания физически нет напечатанных вопросов 19/20
(текст обрывается на следующем варианте). Правило в `prompts.py`
("every required question slot must still get its best-supported
answer from context... rather than omitting the question") напрямую
инструктирует модель придумать что-то, а не оставить вопрос пустым —
и именно так родились полностью выдуманные варианты ответов
("liegt an der Cavone GmbH." и т.д., не существующие в PDF).

Это НЕ одноразовый баг в данных (в отличие от hoeren_teil3/4 варианта
7, которые пофикшены заменой на sentinel) — это структурный конфликт:
"answer" в схеме для questions — обязательное поле (`response_schemas.py`
universal schema, `options` — ровно 3 записи), нет валидного способа
пометить "источник не дал ответа" без ломки формы. Нужно решение на
уровне схемы/промпта: либо разрешить `answer: null` для questions без
маркера в источнике (и научить клиент показывать «нет данных» вместо
вопроса), либо — раз это единичный, не повторяющийся кейс на всём
документе — оставить как есть и просто исключить эту пару вопросов
вручную в courated-патче. Не чинить наспех: тронуть эту UNIVERSAL-схему
затронет ВСЕ 9 типов секций, которые её используют.

## TTS: staged voice_gender rollout (2026-07-16)

Backend `/api/tts` теперь принимает необязательное JSON-поле
`voice_gender: "female" | "male" | "unknown"`. Отсутствующее поле и явное
`"unknown"` сохраняют совместимость: выбор идёт по явной роли speaker
(`Frau`/`Herr` и gendered roles), затем по общему пулу. Явные
`"female"`/`"male"` переопределяют эти эвристики; некорректное значение
возвращает безопасный `400` JSON без raw exception.

Parser/schema могут добавлять optional `metadata` с теми же gender-значениями:
`metadata.voice_gender` для монологов/сообщений и
`metadata.speaker_voice_genders[]` для диалогов с явными speaker labels.
Старые course JSON без metadata остаются валидными. Span resolution для
`lesen_teil2`/`hoeren_teil4` и `telefonnotiz` сохраняет только валидные
metadata-поля и безопасно отбрасывает malformed hints, не меняя legacy
`{title, content}` / `weitere_informationen: [string]` формы. Backend-списки
конкретных имён удалены: новые имена обслуживаются parser metadata, а
сомнительные случаи остаются `unknown` и могут быть исправлены manual override.

Flutter-клиент поддерживает этот staged контракт: `VoiceGender` defensively
парсит metadata, manual override имеет приоритет над parser hints, затем идёт
explicit `Frau`/`Herr`, затем `unknown`. TTS cache key переведён на v2 с
gender-компонентом, чтобы исправленный выбор голоса не переиспользовал старый
MP3. Parse cache поднят до `v37`. UID-isolated voice preferences хранятся в
SharedPreferences с latest-wins записью, не создаются без UID и очищаются при
account deletion. Ключ также включает course ID и устойчивую позицию записи.

UI показывает localized Automatic/Female/Male controls, включая per-speaker
controls для диалогов. Переключение text/recording/metadata/override в
`DialogueAudioPlayer` останавливает playback, invalidates stale operations,
releases typed leases, reparses lines and stays idle (no autoplay). Stable
recording IDs есть для Telefonnotiz editions, Hören Teil 1 pairs и universal
Hören text cards; duplicate/empty/Cyrillic labels не сталкиваются. Проверено:
Flutter 299/299, coverage 61.33%; backend 86/86 + `py_compile`; device PDF
integration прошёл. Production APK собран, установлен и cold-launched без
crash. Manual TTS listening после авторизации ещё требуется; backend deploy не
выполнялся.

## Cost-aware Promptfoo gate для TTS metadata (2026-07-16)

`promptfoo/run_all.sh` требует явный mode: `--parse-only` запускает 18 cases;

число запросов: Hören Teil 4 проверяет ожидаемую последовательность

### Первый live parse gate и remediation (2026-07-16)

Подтверждённый `--parse-only` pass выполнил ровно 18 Gemini Flash-Lite calls;
discovery и retry не запускались. Фактический usage: 53,050 input, 36,746
candidate-output и 7,777 thinking tokens, всего 97,573; provider оценил
Standard-tier стоимость в `$0.080047`. Результат 14/18 — release заблокирован,
как и должен делать fail-closed gate.

Четыре failures проверены по сохранённому локальному eval без повторного spend:

- Lesen Teil 2 вернул корректные headings непосредственно перед body spans,
  отделённые одной пустой строкой. Runtime теперь детерминированно расширяет
  span на такой соседний heading (не более двух строк, только через blank gap),
  сохраняя fail-closed для дальних/отделённых контентом значений. Captured
  response replay: 4/4 текста разрешаются без sentinel.
- Hören Teil 1 схлопнул шесть `<<<ITEM>>>` fragments в один объект и склеил
  speaker turns. Тип переведён с MINIMAL на LOW thinking; gate теперь требует
  минимум три полные editions, обязательные speaker sets и сохраняет строгие
  exact-label voice checks. Headerless fragments наследуют последний явно
  указанный номер варианта и могут образовать допустимую четвёртую edition.
- Telefonnotiz ошибочно превратил `variant 3/1` в `variant_number: 31` и вынес
  edition в отдельный объект. Prompt получил явное правило `N/M → N`; gate
  требует один variant 3 с пятью editions и пятью contextual male hints.
- Hören Teil 4 правильно определил фактического male narrator вместо
  отсутствующей Frau Plassberg, но ошибся для явно указанной Frau Bernhardt.
  Voice rule теперь требует сначала определить фактического говорящего по
  self-identification и matching question/answer block. Fixture expectation:
  `female, male, female, male, female`; конкретные имена в production prompt
  не добавлялись.

После remediation выполнялись только change-focused checks. Telefonnotiz стал
зелёным; H4 исправляется детерминированно по same-speaker `Frau/Herr` evidence
и прошёл replay трёх оплаченных outputs; H1 сохраняет speaker turns и даёт 3
обязательные полные editions, иногда плюс четвёртую headerless edition. Оба
варианта допустимы: минимум 3, один явный source variant number, три required
speaker sets, optional `Chef/Frau`, без placeholder content.

Последняя попытка offline replay через Promptfoo `--model-outputs` неожиданно
не отключила custom Python provider: Promptfoo автоматически загрузил локальный
`.env` и сделал ещё один полный live run. Не использовать этот флаг как
гарантию отсутствия spend для данного config. Этот run дал 17/18; единственный
H1 failure был старым чрезмерным exact-count=4. Его сохранённый output содержит
3 полные editions и проходит текущие assertions, поэтому regrade текущего
последнего набора — эффективные 18/18 без нового API-вызова.

Суммарная оценённая стоимость всех содержательных live calls этой remediation:
`$0.222686`. Две попытки H1 на 3.5 Flash завершились 503/timeout до usage и
показали `$0`; сильная модель отменена, H1 остаётся на Flash-Lite + LOW.
Production deploy, cache migration, commit и push на момент этой записи ещё не
выполнялись.

### CR-07 delete-wins course sync — 2026-07-18

повторный импорт создаёт новый UUID. `courses/{uid}/items/{courseId}` теперь

идемпотентен; stale/legacy POST tombstone не воскрешает.

updatedAt}]`; tombstones не попадают в `courses`. Firestore list failure теперь
503, не fake-empty 200. Course IDs ограничены безопасным форматом, а исторический
JSON выдаётся только при совпадении `course.id` с Firestore document id. CORS
