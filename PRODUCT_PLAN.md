# План: публичный продукт — минимум затрат на ИИ, максимум качества

Дата: 2026-07-12. Контекст: приложение публикуется и монетизируется;
модель «пользователь загружает свой PDF» остаётся (юридическая позиция:
продаём инструмент, контент приносит пользователь). **Оба ключа Gemini
теперь платные** (free-тариф больше не сидит на бесплатной квоте) — это
меняет приоритеты: защита от расходов нужна и для free-тарифа.

Опорные факты по стоимости (замерено сегодня на реальном документе):

| Вызов | Модель | Токены | Цена |
|---|---|---|---|
| discover (1 раз на новый документ) | gemini-3.5-flash | ~154K in + ~15K out/thoughts | ~$0.35 |
| parse (все 12 секций суммарно) | gemini-3.1-flash-lite | мелкие чанки | ~$0.10–0.20 |
| **Итого первый импорт нового PDF** | | | **~$0.50** |
| Повторный импорт того же PDF (кэш) | — | 0 | **$0** |

Кэш глобальный (ключ = хэш содержимого, без TTL, не привязан к
пользователю). Ниша узкая — большинство пользователей импортируют одну и
ту же компиляцию из Telegram-канала. Значит предельная стоимость
пользователя ≈ $0, платим только за «первопроходца» каждой версии PDF —
если закрыть дыры для злоупотреблений (Фаза 0) и перестать самим сбрасывать
кэш зря (уже починено: раздельные версии discover/parse кэша).

## Фаза 0 — защита кошелька (СНАЧАЛА, до публикации; ~полдня)

Сейчас rate limit — 1000 запросов/час на пользователя. Один discover
≈ $0.35 → злонамеренный аккаунт с уникальными PDF может сжечь сотни
долларов за час. Оба тарифа на платных ключах, поэтому:

### 0.1 Лимит импортов (discover-вызовов) в день на пользователя
- В `main.py::parse()` для `section_type == 'discover'` (только при
  кэш-промахе — кэш-хиты не считаются): Redis-счётчик
  `importcap|{uid}|{YYYYMMDD}`, expire 24ч.
- Premium: **5 новых документов/день** (реальному студенту хватает с
  запасом; потолок расходов $2.5/день на аккаунт).
- Free: см. 0.2.
- Ответ при превышении: 429 + понятное сообщение (клиент уже умеет
  показывать серверные ошибки лимитов).

### 0.2 Free-тариф: discover только по кэшу (рекомендуемый вариант)
Free-пользователь НЕ может запустить discover для документа, которого
нет в кэше — вместо этого получает сообщение «этот документ ещё не
обработан — доступно в Premium». Следствия:
- Free-тариф теперь стоит **строго $0** в Gemini-расходах на discover
  (parse для free и так ограничен 12K символов / 8 изданий — копейки,
  плюс тоже можно закрыть той же логикой «только по кэшу»).
- Популярную компиляцию зальёт первый же premium (или мы сами через
  Фазу 1) — free-пользователи получат полный опыт на главном контенте.
- Незнакомый PDF становится апселлом в premium — честная и понятная
  граница тарифа.
- Реализация: в `parse()` при кэш-промахе и `not premium` → 402/403 с
  кодом ошибки для клиента. Альтернатива (если вариант покажется
  жёстким): 1 импорт/день free — но это до $0.5/день на каждый free-
  аккаунт, при масштабе это дыра; вариант «только по кэшу» безопаснее.

### 0.3 Глобальный дневной предохранитель
Redis-счётчик discover-вызовов на весь сервис в день (например, 100).
При превышении новые документы получают «сервис перегружен, попробуйте
завтра», кэш-хиты продолжают работать. Защищает от координированного
злоупотребления множеством аккаунтов. Констант в одном месте, рядом с
`_RATE_LIMIT_PER_HOUR`.

### 0.4 Мониторинг расходов
- Еженедельно: `vercel logs ... | python3 scripts/cost_report.py`
  (уже считает по-модельно честно).
- В GEMINI_USAGE лог добавить `uid=` (усечённый) — чтобы при аномалии
  видеть, кто именно жжёт токены.

**Definition of done Фазы 0:** злонамеренный аккаунт любого тарифа не
может потратить больше $2.5/день; free-тариф стоит ~$0; есть глобальный
стоп-кран.

## Фаза 1 — курируемый флагманский контент через прогрев кэша (~1 день)

Идея: воркфлоу сверки (офлайн-парсинг → `verify_content.py` → ручные
правки) производит эталонный JSON, который **записывается в Redis под
теми же ключами, которые клиент вычислил бы сам**. Пользователь
импортирует популярный PDF → хэш совпал → мгновенно получает выверенную
версию за $0. В приложении и API не меняется ни строчки.

### 1.1 `tools/inject_curated.py`
- Вход: markdown источника + курируемый course JSON (+ версии кэша).
- Пишет **один ключ уровня `doc`** (кэш целого курса —
  `getCachedSections` проверяет его первым, при хите discover/parse
  вообще не вызываются): `{discoverVer}.{parseVer}|doc|sha256('doc|'+markdown)`.
- Порт хэширования из `parse_service.dart` (sha256 utf-8, формат входа
  `doc|<markdown>`); обязательная самопроверка: прочитать из Redis
  существующий ключ, записанный клиентом, и убедиться, что вычисленный
  скриптом ключ совпадает байт-в-байт.
- **Критичная деталь:** markdown должен быть байт-идентичен тому, что
  выдаёт продовый `/api/convert` (в requirements.txt markitdown==0.1.0,
  локально стоит 0.1.6 — вывод РЕАЛЬНО отличается: подтверждено
  2026-07-13, хэш локальной 0.1.6-конвертации не совпал с ключом в
  Redis, работа по инъекции group-ключей была остановлена на этом).
  Два проверенных способа получить правильный markdown: (а) продовый
  `/api/convert` (нужен Firebase-токен), (б) локальный venv с ровно
  `markitdown[pdf]==0.1.0` — проверено, даёт байт-в-байт серверный
  вывод (sha256 совпал с реальным doc-ключом). Всегда сверять хэш с
  существующим ключом ДО любых вычислений на основе нарезки.

### 1.2 Курируемый контент v1 (по сегодняшнему отчёту сверки)
Основа — `course2.json` (уже вытащен с телефона). Исправить
подтверждённое:
- telefonnotiz: 3 случая обрезания `weitere_informationen` по «/»
  (восстановить полный текст буллетов из источника);
- telefonnotiz: «вариант №3/1» — сейчас это отдельный item с
  variant_number=31; переделать в издание варианта 3 (как пайплайн сам
  сделал с «№1/1»);
- hoeren_teil4 вариант 8: удалить галлюцинированные транскрипты и
  выдуманные ключи ответов (в источнике их нет) — оставить честное
  «(nicht angegeben)» / пометку «без ответов в источнике»;
- beschwerde вариант 1, Q19: ключ b → c (проверено по письму Lassner И
  по жёлтой разметке в PDF — стр. 65);
- ВСЕ ключи ответов сверить автоматически по PDF-разметке
  (`scripts/check_answer_keys.py`, см. 2.3.1) — заменяет ручную проверку
  спорных случаев; уже подтверждено разметкой: lesen_teil4 в.2 Q17 обе
  редакции извлечены неверно;
- разобрать lesen_teil2 (6+ items с перепутанными текстами/вопросами —
  самое серьёзное);
- перегнать сверку hoeren_teil2 (сегодня упал с 503).

### 1.3 Правки — в `overrides.json`, а не в сгенерированный файл
Формат: список патчей `{path, old, new, reason}` + скрипт
`tools/apply_overrides.py`. Пересборка курса при обновлении PDF не
теряет ручные исправления; каждый патч самодокументирован.

### 1.4 Правило поддержки
При каждом бампе версии кэша — перезалить курируемые записи
(`inject_curated.py` под новую версию). Одна строка в чек-лист релиза.

**Definition of done Фазы 1:** импорт флагманского PDF любым
пользователем возвращает выверенный курс мгновенно и бесплатно; ручные
правки переживают пересборку.

## Фаза 2 — детерминизация пайплайна для незнакомых PDF (инкрементально, 1–2 недели)

Для «длинного хвоста» чужих документов. Каждый шаг проходит через
promptfoo-гейт (`run_all.sh`) с новыми regression-фикстурами.

### 2.1 Line-span для verbatim-полей
Сейчас модель ПЕРЕПЕЧАТЫВАЕТ дословные тексты (монологи, письма,
транскрипты) — отсюда перефразирование, галлюцинации и большие
output-токены (самые дорогие). Переход: чанк отправляется с
пронумерованными строками (как уже сделано в discover), схема возвращает
`{start_line, end_line}` вместо текста, клиент вырезает текст кодом.
- Перефразирование/галлюцинация на этих полях становятся структурно
  невозможными; output-токены падают в разы → дешевле.
- Порядок внедрения по плотности багов: telefonnotiz (буллеты!) →
  lesen_teil2 (перепутанные тексты) → hoeren_teil4 (галлюцинация) →
  остальные.
- На каждый тип: схема + промпт + клиентская вырезка + фикстура + бамп
  `_parseCacheVersion` (+ перезаливка курируемого, см. 1.4).

### 2.2 Ключи ответов регэкспом, где есть маркер
Источник помечает правильные ответы маркерами «– 100%» / «-100%» /
«(100%)» / буквой после вопроса. После line-span у клиента есть точный
текст — извлекать ключ регэкспом БЕЗ LLM. В JSON добавить
`answer_source: "marker" | "model"`:
- «marker» — детерминированный, доверяем;
- «model» — вопросы на понимание текста (beschwerde Q19/20, lesen_teil4
  Q14–18 — Protokoll/письма, маркеров в источнике нет) — кандидаты на
  ручную проверку в курируемом контенте, UI может помечать их менее
  жирной уверенностью.

### 2.3 Извлечение ключей из PDF-разметки (highlights/underline)
**Открытие 2026-07-12 (проверено на реальном PDF):** правильные ответы
для ВСЕХ типов вопросов физически размечены в PDF жёлтой заливкой +
подчёркиванием — включая beschwerde и lesen_teil4, где текстового
маркера «– 100%» нет. MarkItDown выбрасывает форматирование, поэтому в
markdown ключ отсутствует и модель вынуждена угадывать — отсюда все
сегодняшние «неверные ключи». Это не задача рассуждения, а потеря
данных на этапе convert. Никакой сильной модели не нужно.

Извлечение (подтверждено PyMuPDF на стр. 65 и 47/49): жёлтые filled-
rect'ы (fill 1,1,0, высота >5pt) поверх строк опций; `page.get_text
(clip=rect)` возвращает точный текст опции. Семантика цветов: жёлтый на
строке-опции = ключ ответа; жёлтый на заголовке = просто выделение
заголовка; зелёный = мета-метки («Новый вариант от ...») — фильтровать
по паттерну строки-опции `^[a-c]\)`.

Реализация в два этапа:
1. **Офлайн сразу** (СДЕЛАНО 2026-07-12): `scripts/check_answer_keys.py`
   — вытаскивает все подсвеченные опции из PDF и детерминированно
   сверяет каждый ключ в course JSON. Результат первого прогона по
   актуальному курсу: **257 вопросов → 222 OK, 21 MISMATCH (~8% ключей
   неверны!), 13 без разметки в источнике, 1 неоднозначный.** Все
   ~9 «спорных» заявок LLM-аудитора подтвердились + найдено ещё ~12,
   которые аудитор пропустил (hoeren_teil3 в.7, hoeren_teil4 в.6/7/8,
   lesen_teil4 в.3/6). Полный список: scratchpad
   `courses/answer_keys_audit.txt` → все 21 идут в overrides
   курируемого контента (Фаза 1.2).
2. **В рантайм-convert позже**: post-pass после MarkItDown — найти
   подсвеченные сниппеты в markdown (нормализованный поиск строки, та же
   техника, что anchor-коррекция) и дописать существующий маркер
   «– 100%» к строкам-опциям. Все текущие правила промптов уже доверяют
   этому маркеру — они продолжают работать без изменений; regex-
   извлечение из 2.2 начинает покрывать и beschwerde/lesen_teil4.
   Требует: pymupdf в requirements (проверить лимит размера функции
   Vercel, wheel ~50MB), бамп версии кэша, promptfoo-гейт.

### 2.4 Нормализация «вариант №X/Y»
Решить семантику один раз: «X/Y» = под-издание варианта X (так пайплайн
сам обработал «1/1»). Правило в промпт + фикстура. variant_number
остаётся INTEGER (X), «/Y» уходит в label.

**Definition of done Фазы 2:** verbatim-поля никогда не расходятся с
источником; ключи с маркером извлекаются без LLM; comprehension-ключи на
сильной модели; promptfoo зелёный.
